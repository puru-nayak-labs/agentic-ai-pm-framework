North Star & ROI Framework
★ North Star Definition

What does success look like — measurably — in 6 months?

Before any gate, define the single metric that will tell you the agent is working. Every downstream decision flows from this. If you cannot write it in one sentence, pause here.

ROI Inputs — Capture Before Building
InputHow to Measure
Human time per taskTime study or task tracking data
Tasks per day / weekTicket system, process logs
Fully-loaded human costSalary + overhead per hour × time
Current human error rateQA records, rework logs
Cost of one agent errorDefine with legal, finance, ops
Estimated agent cost/taskTokens × model price + infra
ROI Decision Rule

Go / No-Go Thresholds

  • Agent cost <40% of human cost → strong go
  • Agent cost 40–80% of human cost → go if quality gain justifies
  • Agent cost >80% of human cost → rethink scope
  • Volume <50 tasks/week → ROI rarely justifies
  • Error cost exceeds monthly savings → redesign oversight first
Load-Bearing AI Test from 0→1 playbook
★ Critical gate before committing to agent architecture

Ask one question: Does the agent's removal make the core outcome impossible — or just slower? If impossible → load-bearing AI, build the agent. If slower → simpler LLM feature embedded in existing product. Only agentic systems justify the architecture, governance, and maintenance cost of a full multi-step agent build.

Pre-Build Checklist
Complete before proceeding to Stage 2
  • North Star metric written in one sentence — agreed by all stakeholders
  • ROI inputs captured: human time, volume, cost per task, error rate, error cost
  • Load-bearing AI test passed: agent removal makes outcome impossible, not just slower
  • ROI go/no-go threshold evaluated and documented
  • Workflow mapped step-by-step including every decision fork
  • Data sources identified, access confirmed, format assessed
  • Success criterion written — specific, measurable, evaluatable
  • Human override path designed
  • Compliance and regulatory constraints identified
  • Owner of outcome when agent is wrong: named
  • Concierge MVP completed: outcome delivered manually to 5+ users who acted on it
Key Questions to Ask Stakeholders

Walk me through the last 5 times this task was done. Where did it take longest? Where did something go wrong?

Real failure modes surface here that requirements docs never capture.

If the agent gets it wrong 1 in 20 times — what happens downstream, who finds out, and how fast?

Forces explicit error cost definition before anyone writes code.

What does the data look like right now — queryable, or in emails, PDFs, and spreadsheets?

Data access is the most common silent blocker at this stage.

In 6 months, what single number tells us the agent is working? What do we watch weekly to know we're on track?

Separates vanity metrics from actionable leading indicators.

Who owns the outcome when the agent is wrong — are they in this room right now?

Accountability gap here becomes a production crisis later.

If we delivered this outcome manually to 10 users this week — would they act on it? Have we tested that?

The Concierge MVP question. If the answer is no or untested, don't build the agent yet.

Does the product fail without AI — or is AI making an existing process faster?

The load-bearing test. Only build agents for the former.

Every architectural choice maps to a cost, latency, or quality tradeoff

6 Architectural Dimensions
Dimension A

Reasoning & Planning

  • Chain-of-thought vs ReAct vs Tree-of-Thought
  • Multi-step goal decomposition
  • Planning depth vs latency tradeoff
  • Reflection and self-critique loop
  • Hierarchical agent patterns
Dimension B

Memory Architecture

  • In-context memory scope
  • External long-term: vector DB selection
  • Episodic vs semantic vs procedural
  • Memory retention policy
  • Multi-session continuity strategy
Dimension C

Tool Use & Actions

  • Tool taxonomy: read / write / execute by risk
  • Permission boundaries per tool class
  • Human approval checkpoints
  • Action verification and rollback paths
  • Parallel vs sequential orchestration
Dimension D

Orchestration Pattern

  • Single agent vs multi-agent
  • Orchestrator + sub-agent pattern
  • Human-in-the-loop placement
  • Failure propagation isolation
  • Inter-agent communication protocol
Dimension E

Context Strategy

  • Prompt compression and summarization
  • Prompt caching (major cost lever)
  • Token budget per sub-task
  • Context overflow handling
  • System prompt vs dynamic injection
Dimension F

Governance by Design

  • PII detection at input AND output
  • Prompt injection defense
  • RBAC for agent action classes
  • Full audit trail for every action
  • Kill switch and graceful degradation
Key Questions

Which model tier handles which sub-task — routing or defaulting frontier model everywhere?

Routing is the highest-ROI architecture decision.

Where are the human approval gates — fast enough not to negate the value of automation?

A 4-hour human gate breaks an async agent.

How does the system degrade when a tool call fails or context is insufficient?

Failure paths need as much design as success paths.

Prototype Build Checklist
Must be in place before declaring prototype complete
  • System prompt: role / context / constraints / output format / tool descriptions
  • Prompt versioning active — every change tracked with version ID
  • Agent loop: Observe → Plan → Act → Reflect
  • Max iteration cap set — no infinite loops possible
  • Confidence threshold for irreversible actions defined
  • At least 20 manual test cases run with results logged
  • Silent failure modes identified with detection mechanism for each

AI Leading Metrics from 0→1 playbook
★ Leading vs. lagging — track both, but act on leading

Lagging metrics (revenue, churn, NPS) tell you what happened. Leading metrics tell you what's about to happen. For agentic systems: AI action rate (% of tasks where agent takes autonomous action without triggering override) and agent override rate trend (week-over-week direction) are your most reliable early signals of whether the system is earning trust or losing it. If override rate is rising before users churn, you have 2–3 weeks to fix it.

3-Layer Eval Framework
Layer 1 · Offline

Before Anything Ships

  • Golden dataset: 100+ real edge cases
  • Prompt regression tests on every change
  • Benchmark against human baseline
  • Tool selection accuracy scored
Layer 2 · Online

In Production, Always

  • A/B prompt variant testing
  • Human feedback capture per session
  • LLM-as-judge production scoring
  • AI action rate and override trend weekly
Layer 3 · Agent-Specific

Agentic Metrics Only

  • Multi-step goal completion rate
  • Tool calls per successful task
  • Recovery quality after mid-task failure
  • Cost per verified successful outcome
Key Questions

Is the golden dataset built from real edge cases and failures — or just happy paths that worked in demos?

Datasets built from successes measure the wrong thing.

What is the regression gate that must pass before any prompt change reaches production?

No prompt change ships without running the full eval suite.

Is your AI action rate rising week-over-week — and is override rate declining?

These are the two leading indicators that your agent is earning trust, not just processing tasks.

Production Readiness Checklists
Security & Governance — Before Go-Live
  • Prompt injection tested across all input surfaces
  • PII detection at input AND output layers
  • RBAC enforced per user role on all tool classes
  • Full audit trail confirmed end-to-end
  • Data residency compliance verified
  • Kill switch tested in staging
  • Human escalation under 60 seconds tested
Reliability Engineering — Before Go-Live
  • Model API rate limit handling and backoff
  • Tool timeout handling and retry logic tested
  • Context window overflow graceful truncation
  • Infinite loop prevention: max-turn caps
  • Partial completion state recovery tested
  • Circuit breakers on all external tool calls
  • Prompt rollback procedure rehearsed
Key Questions

If the model API goes down mid-execution — does the task resume, fail gracefully, or corrupt state?

Partial failure will happen. Design for it explicitly.

If a bad prompt change reaches production, what is the rollback procedure and how long does it take?

Should be as fast and practiced as a code rollback.

6 Mechanisms of Self-Improvement
Mechanism 1

Feedback Capture

  • User corrections → labeled failure cases
  • Override events → failure library
  • Task abandonment as implicit negative signal
  • A/B winner signals → prompt queue
Mechanism 2

Reflection & Self-Critique

  • ReAct observation after each action
  • Constitutional AI self-evaluation
  • Cross-agent verification
Mechanism 3

Prompt Auto-Optimization

  • DSPy automated prompt optimization
  • Failure replay → targeted prompt patches
  • Few-shot refresh from recent successes
  • A/B tests with significance gates
Mechanism 4

Memory Personalization

  • User preference learning
  • Org knowledge base continuous updates
  • Retrieval ranking tuned from corrections
  • Episodic memory pruning
Mechanism 5

Synthetic Eval Generation

  • Failed tasks → auto regression cases
  • Edge case synthesis from failure taxonomy
  • Red-team agents generating adversarial examples
  • Golden dataset grows from every failure
Mechanism 6

Fine-Tuning Pipeline

  • RLHF from curated preference data
  • RLAIF: AI feedback to scale labeling
  • LoRA adapters for task-specific behavior
  • Continuous fine-tuning with drift detection
Data Flywheel & Moat from 0→1 playbook
★ Self-improvement is your moat — build it from day one

A self-improving agent is the ultimate data flywheel: each production run generates signals that make the next run better. Combined with workflow integration (switching cost) and network effects from multi-user memory, this creates a compounding competitive advantage that a newer agent without your data cannot replicate. The moat is not the model — it is the feedback loop architecture built into the system from Stage 1. Teams that design self-improvement as an afterthought lose this entirely.

Governance Checklist — Required Before Any Autonomous Improvement Ships
Non-negotiable governance gates
  • Every production failure automatically generates a regression test case
  • Circuit breaker defined: conditions that pause the improvement loop
  • Human approval gate required before any self-generated prompt update reaches production
  • RLAIF judge model calibrated against human ground truth
  • Drift detection active: alerts when behavior shifts outside defined bounds
  • Synthetic evals reviewed before entering golden dataset
Key Questions

When the agent fails today, does that failure automatically become a test case preventing the same failure tomorrow?

If failures don't generate evals, the same tuition gets paid twice.

How do you detect when the self-improvement loop is making things worse — and what triggers the circuit breaker?

Autonomous improvement without guardrails is autonomous drift.

Is your self-improvement loop generating compounding data advantage — or are you just logging without acting on it?

The moat is built from the feedback loop, not the model. Act on every signal.

The goal is a system that treats every failure as a lesson, every correction as a label, and every day in production as a training run — within governance guardrails designed from day one. That is not just reliability. That is the moat.
